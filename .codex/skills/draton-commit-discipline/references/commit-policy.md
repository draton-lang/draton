# Commit policy

## Core rules

- Create a commit after every meaningful change.
- Commit as soon as the code or docs are in a stable, runnable state.
- Keep each commit to one logical unit of work.
- Use clear, specific commit messages.
- Avoid vague messages such as `fix`, `update`, `wip`, `temp`, or `misc`.

## Default format

```text
<type>: <short summary>
```

## Example types

- `feat`
- `fix`
- `refactor`
- `docs`
- `test`
